# Requirement writing

Use `REQ-<DOMAIN>-NNN` for functional requirements and `NFR-<DOMAIN>-NNN` for quality requirements.
Keep IDs unique, stable across editorial changes, and never recycle removed IDs.

Write each requirement with one observable obligation. Identify the actor/system, trigger or
precondition, behavior, result, important error behavior, and acceptance criteria when confirmed.
Use MUST only for confirmed obligations. Keep recommendations, open options, and assumptions in
explicitly labelled sections instead of normative requirements.

Do not add a numeric threshold, retention period, permission, workflow transition, integration
behavior, or compliance rule unless it appears in the source or a recorded answer.
